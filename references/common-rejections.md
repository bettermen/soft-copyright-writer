# 常见驳回原因与避坑策略（2026年版）

## 一、源代码相关问题（占比最高，约 40%）

### 1. 代码量不合规
- **现象**：每页不足 50 行、总页数不足
- **原因**：空行和注释未去除，或代码量确实不足
- **避坑**：提交前严格去除空行和纯注释行，统计有效行数；不足时合并短文件

### 2. 包含非原创代码
- **现象**：代码查重率过高
- **原因**：包含了开源框架代码、第三方库、自动生成代码
- **避坑**：只提交核心业务逻辑代码，排除 node_modules、vendor、框架源码
- **2026 新变化**：审查系统已升级查重算法，自动检测与 GitHub 开源项目的相似度

### 3. 页眉页脚不规范
- **现象**：缺少页眉、页脚版本号错误
- **原因**：忘记添加或模板遗漏
- **避坑**：每页必须含「软件全称 V{版本号}」页眉和「第 X 页 / 共 Y 页」页脚

### 4. 代码与文档不一致
- **现象**：文档描述的功能在代码中找不到对应实现
- **原因**：文档写得超前，代码还没实现；或用了不同版本的代码
- **避坑**：按照实际代码逻辑撰写说明书，不要凭空添加功能

### 5. 代码含敏感信息
- **现象**：代码中暴露数据库密码、API Key 等
- **原因**：硬编码了敏感配置
- **避坑**：将敏感信息替换为占位符（如 `DB_PASSWORD = "***"` 或 `API_KEY = "YOUR_KEY_HERE"`）

---

## 二、设计说明书相关问题（约 30%）

### 1. 字数不足
- **现象**：全文不足 3000 字
- **原因**：功能描述过于简略，模块拆分不够
- **避坑**：每个功能模块至少写 500-800 字，3 个模块即可达标

### 2. 内容与代码不符（「名不副实」）
- **现象**：文档描述「AI 智能分析」功能，代码中无 AI 模块
- **原因**：为了凑字数或包装功能而夸大描述
- **避坑**：只描述代码中实际存在的功能，功能描述要具体化

### 3. 大段复制网络内容
- **现象**：内容与网上文章雷同
- **原因**：为了凑字数直接复制粘贴
- **避坑**：基于本软件的实际功能撰写原创内容

### 4. 软件名称/版本号与申请表不一致
- **现象**：文档标题和申请表填的不一样
- **原因**：修改后未同步更新
- **避坑**：提交前全文搜索软件名称和版本号，确保一致

### 5. 缺少关键章节
- **现象**：没有运行环境/没有操作流程
- **原因**：模板不完整
- **避坑**：至少包含：引言、总体设计、功能详述（3+ 模块）、运行说明

## 三、申请表相关问题（约 20%）

### 1. 软件名称不规范
- **现象**：含版本号、是通用词汇、有敏感词
- **避坑**：遵循「品牌+功能+后缀」格式，提交前官网查重

### 2. 日期逻辑错误
- **现象**：开发日期 > 申请日期，首次发表 < 开发日期
- **避坑**：确保时间线合理：开发完成 → 首次发表 → 申请登记

### 3. 申请主体信息不一致
- **现象**：企业名称与营业执照不一致
- **避坑**：严格按营业执照全称填写

### 4. 源代码行数填写不实
- **现象**：填了 10 万行，实际提交的代码只有 3000 行
- **避坑**：填写与提交源代码文档一致的统计行数

---

## 四、身份证明材料问题（约 10%）

### 1. 证照模糊/过期
- **现象**：扫描件不清晰、信息不可辨识
- **避坑**：300 DPI 以上扫描，确保文字清晰

### 2. 缺少必要授权文件
- **现象**：企业申请没有职务作品授权书
- **避坑**：企业申请必须准备职务作品授权书

### 3. 合作开发缺少协议
- **现象**：多个申请人但无合作协议
- **避坑**：合作开发必须提交署名合作协议

---

## 2026 年审查新趋势

### 更严格的「非正常申请」审查
- 短时间内集中提交大量相似软著 → 可能被标记
- 代码与开源项目高度一致 → 直接驳回
- 委托代理机构「包过」但材料造假 → 列入黑名单

### 文档-代码一致性审查强化
- 审查员会比对文档功能描述与代码实现
- 「名不副实」成为高频驳回理由

### 电子证书全面推行
- 2026 年起只发电子证书，不再邮寄纸质版
- 通过官方系统自行下载打印

---

## 提交前最终检查清单

```
□ 源代码每页 ≥ 50 行有效代码（已去空行和注释）
□ 源代码页眉含软件全称和版本号
□ 源代码为原创核心模块，无第三方库
□ 设计说明书 ≥ 5000 字
□ 设计说明书中软件名称、版本号与申请表一致
□ 设计说明书中描述的功能在代码中有对应实现
□ 申请表软件名称符合规范，已查重
□ 申请表日期逻辑正确
□ 身份证明清晰有效
□ 所有文件已转 PDF
□ 所有文件单个大小 ≤ 20MB
```
